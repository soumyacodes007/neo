# Volume 14 - Browser Companion Wallet Bridge

This volume specifies the browser-based approval window for Phase 8. It explains how the window talks to the local MCP server, how smart-account-kit fits, what the tech stack should be, and which tests prove the bridge is safe enough for the SCF submission.

The target is Claude Desktop plus a local MCP server. The same design also works for other MCP hosts because the signing surface is only a loopback browser page plus signed XDR handoff.

## 1. Research Basis

Verified on 2026-07-06:

- WebAuthn/passkey ceremonies are browser/native-platform operations and require a secure context. Loopback origins such as `http://localhost` and `http://127.0.0.1` are treated as potentially trustworthy by the secure-contexts model, which is why local development passkey pages work without HTTPS.
- The OAuth native-app loopback pattern is standardized in RFC 8252: a local app starts an ephemeral loopback listener, opens the system browser, and receives a one-time callback on the local listener. We reuse this topology; we are not implementing OAuth.
- smart-account-kit is the right Stellar/OZ smart-account SDK layer. It provides create/connect wallet behavior, context-rule management, auth payload/sign/submit helpers, passkey and Ed25519 signer paths, and credential storage concepts. Exact API calls must be pinned to the package version during implementation.
- The MCP server should remain the deterministic tool boundary. The browser companion is only the owner-approval and signing surface. F1 remains the state-changing network gate.

Primary references:

- smart-account-kit: https://github.com/kalepail/smart-account-kit
- MCP TypeScript SDK: https://github.com/modelcontextprotocol/typescript-sdk
- WebAuthn API: https://developer.mozilla.org/en-US/docs/Web/API/Web_Authentication_API
- Secure contexts: https://developer.mozilla.org/en-US/docs/Web/Security/Secure_Contexts
- W3C potentially trustworthy origins: https://w3c.github.io/webappsec-secure-contexts/#potentially-trustworthy-origin
- OAuth native app loopback pattern: https://www.rfc-editor.org/rfc/rfc8252.html

## 2. Decision

We will use a top-level browser companion window for owner approval and passkey signing.

The MCP server will run a local loopback HTTP bridge bound to `127.0.0.1` on an ephemeral port. When the user must create a smart account or sign an install plan, the MCP returns an approval URL. Claude Desktop shows the URL, and the skill can ask the host or OS to open it. The user approves in the normal browser window. The browser signs with smart-account-kit and posts signed XDR back to the loopback bridge.

The MCP never receives owner private keys, passkey secrets, wallet passwords, or raw WebAuthn credentials. It receives only account metadata, public signer data, and signed XDR.

## 4. Trust Boundaries

Owner credential:

- Lives only in the browser/native authenticator or local encrypted wallet store.
- Used by smart-account-kit to sign exact transactions or auth payloads.
- Never crosses loopback.

Session signer:

- A separate Ed25519 grantee key used after the owner installs a scoped rule.
- May be generated by the MCP for the demo, but only as an explicitly approved, scoped, expiring capability key.
- Must never be confused with the owner key.
- Should be stored encrypted at rest or in a session workspace with strict permissions for the first demo.

MCP server:

- Trusted to verify, plan, and submit through F1.
- Not trusted with owner keys.
- Must treat every browser callback as untrusted input until decoded and matched to the pending request.

Browser companion:

- Trusted only to hold the owner signing ceremony.
- Not trusted to choose what to sign. The MCP supplies exact unsigned XDR and plan hash; the bridge and F1 verify returned XDR.

AI model:

- Sees tool outputs and URLs.
- Must not see private keys, wallet passwords, raw approval token values, or bridge bearer tokens in structured tool output.
- Can relay an approval URL, but cannot produce a valid signed XDR without the user's signer.

## 6. Recommended Tech Stack

MCP server:

- Runtime: Node 22.
- Language: TypeScript, ESM.
- MCP: `@modelcontextprotocol/sdk`.
- Transport: stdio first; streamable HTTP later only if needed.
- Validation: existing zod schemas from `packages/core`.
- Logging: structured JSON lines, no request bodies containing tokens/XDR unless explicitly redacted.

Loopback bridge:

- Package: `packages/wallet-bridge`.
- HTTP server: Node built-in `node:http`.
- No Express/Hono in the first version. The surface is tiny, and fewer dependencies are better for a signing bridge.
- State: in-memory request map with TTL; disk only for already existing plan artifacts, never for bridge bearer tokens.
- Crypto: Node `crypto.randomBytes`, `crypto.timingSafeEqual`, SHA-256 helpers already used by core/stellar packages.

Browser companion:

- Package/app: `apps/companion-web` or `packages/wallet-bridge/src/client`.
- Build: Vite library/static build.
- UI: plain TypeScript plus minimal CSS for first version. React is optional, not required.
- Wallet SDK: pinned `smart-account-kit` version.
- Stellar SDK: pinned to the version compatible with the kit and our XDR code.
- Storage: browser IndexedDB/localStorage only through smart-account-kit or a small adapter. Never browser cookies for private key material.
- Security headers from the loopback server:
  - `Cache-Control: no-store`
  - `Content-Security-Policy: default-src 'self'; connect-src 'self' https://soroban-testnet.stellar.org https://horizon-testnet.stellar.org; img-src 'self' data:; style-src 'self' 'unsafe-inline'; script-src 'self'; base-uri 'none'; frame-ancestors 'none'`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: no-referrer`

Session signer:

- First demo: MCP-generated Ed25519 keypair, encrypted or stored in a session-only workspace with restrictive file permissions.
- Later hardening: companion-owned session key, with MCP requesting signatures through the same bridge.

## 8. Data Model

Signing request:

```ts
type SigningRequestKind =
  | "create_wallet"
  | "connect_wallet"
  | "sign_install_plan"
  | "sign_revocation_plan"
  | "sign_one_off_tx";

interface SigningRequest {
  sid: string;                 // UUID, not secret
  bearerHash: string;          // sha256 of bearer token, raw token never stored
  kind: SigningRequestKind;
  network: "testnet" | "mainnet";
  planHash?: string;
  account?: string;
  createdAtMs: number;
  expiresAtMs: number;
  status: "pending" | "completed" | "rejected" | "expired";
  payload: SigningPayload;
}

interface SigningPayload {
  humanSummaryMarkdown: string;
  riskSummaryMarkdown: string;
  policyDiffMarkdown: string;
  expectedSigner: {
    account?: string;
    signerKind: "webauthn" | "ed25519" | "delegated";
    verifier?: string;
    publicKeyHint?: string;
  };
  steps: SigningStep[];
}

interface SigningStep {
  order: number;
  stepHash: string;
  unsignedXdr: string;
  description: string;
  networkPassphrase: string;
  authRequirements: unknown[];
}
```

Signing result:

```ts
interface SigningResult {
  sid: string;
  planHash?: string;
  account?: string;
  wallet: {
    sdk: "smart-account-kit";
    sdkVersion: string;
    signerKind: "webauthn" | "ed25519" | "delegated";
    publicSignerRef?: string;
  };
  signedSteps: {
    order: number;
    stepHash: string;
    signedXdr: string;
  }[];
}
```

The bearer token is not the F1 approval token. It is only a browser callback authorization token. The F1 approval token remains the human approval gate stored in the plan artifact flow.

## 9. HTTP Protocol

Start behavior:

1. A tool calls `walletBridge.createSigningRequest(payload)`.
2. The bridge starts if not running.
3. It binds to `127.0.0.1:0`.
4. It returns an approval URL:

```text
http://127.0.0.1:<port>/approve/<sid>#<bearer>
```

The bearer token is in the URL fragment. URL fragments are not sent in the HTTP request line, so the token is not present in access logs. The companion page reads the fragment in browser JavaScript and uses it in `Authorization: Bearer <sid>.<bearer>` for API calls.

Routes:

```text
GET  /approve/:sid
  Serves the static companion HTML bundle.

GET  /api/request/:sid
  Authorization: Bearer <sid>.<bearer>
  Returns the SigningPayload for the pending request.

POST /api/result/:sid
  Authorization: Bearer <sid>.<bearer>
  Accepts SigningResult after browser signing.

POST /api/reject/:sid
  Authorization: Bearer <sid>.<bearer>
  Marks request rejected by user.

GET /healthz
  Local-only health check for tests.
```

Request hardening:

- Bind only to `127.0.0.1`, never `0.0.0.0`.
- Prefer `127.0.0.1` in generated URLs; accept `localhost` only if needed for WebAuthn RP compatibility after testing.
- Reject `Host` headers not equal to the active loopback origin.
- Reject requests with an `Origin` header not equal to the active loopback origin.
- Disable CORS by default.
- Use `POST` plus `Content-Type: application/json` for result/reject.
- Maximum JSON body size: 1 MiB by default, configurable for large multi-step plans.
- TTL: 10 minutes default.
- Single use: a completed/rejected/expired request cannot be reused.
- Rate limit invalid bearer attempts per `sid`.

## 10. Exact Flow: Create Wallet

```text
User: "set up my smart account"
Claude skill -> MCP tool: create-wallet-request
MCP:
  starts bridge
  creates SigningRequest(kind=create_wallet)
  returns approval URL
User:
  opens browser window
Browser:
  loads companion page from 127.0.0.1
  calls GET /api/request
  shows network, account type, funding mode, and signer type
  calls smart-account-kit createWallet/connectWallet flow
  passkey ceremony happens in browser/native authenticator
  deploy/connect smart account on testnet
  POSTs account address + public owner signer metadata to bridge
MCP:
  verifies returned account shape
  runs inspect-account
  stores account address in workspace config
  returns account address to the skill
```

Wallet creation does not grant the agent any spend authority. It only creates or connects the user-owned smart account.

## 11. Exact Flow: Sign And Install A Policy

```text
Pipeline:
  trace-transaction
  extract-auth-contexts
  parse-intent
  synthesize-ruleset
  match-policies
  generate-tests
  run-simulation
  detect-bypass
  prepare-install-plan
  explain-policy

MCP:
  creates SigningRequest(kind=sign_install_plan)
  payload includes:
    plan_hash
    deterministic summary
    policy diff
    risk report
    unsigned XDR steps
    expected account
    expected network
    expected owner signer class
  returns approval URL

Browser:
  loads exact plan
  independently shows plan_hash and step count
  user reviews
  user clicks Approve and Sign
  smart-account-kit signs the exact unsigned XDR/auth payloads
  browser POSTs signed XDRs to /api/result

Bridge:
  verifies signed result matches the pending request
  resolves waiting MCP request

F1 submit-plan:
  re-loads plan artifacts from disk
  re-computes plan hash chain
  checks human approval token
  checks network and live account state
  checks signed XDR semantic equality and auth expiry
  submits direct or relayer
  reads back installed rule with inspect-account/inspect-rule
```

The browser does not decide what policy to install. It can only sign or reject the plan the MCP already generated and verified.

## 12. Exact Flow: Silent Repeat With Session Key

After install, the context rule contains the session signer's public key and policies that enforce the scoped behavior.

```text
User asks for same covered action
MCP:
  builds the transaction
  inspects account rules
  detects matching installed context rule
  computes AuthPayload digest
  signs with scoped session Ed25519 key
  submits through F1-style preflight or a lower-risk use path
```

If the action is not covered, the session key signature fails or no matching rule is found. The skill must route back to the owner approval flow.

Session key rules:

- The session key is not the owner key.
- The session key must be short-lived.
- The revocation plan must be generated at install time.
- Any session-key storage location must be displayed in the risk report.
- Production hardening should move this key to the companion wallet or OS keychain.

## 13. Exact Flow: Revoke

```text
User clicks revoke or asks to revoke
MCP:
  prepares or loads revocation plan
  runs preflight and simulation
  creates SigningRequest(kind=sign_revocation_plan)
Browser:
  shows rule being expired/removed
  owner signs revocation XDR
Bridge:
  returns signed XDR
F1:
  verifies token/hash/network/live state
  submits
  reads back that the rule is expired/removed
```

Revocation uses the same owner approval path as installation.

## 14. How We Know It Is The Same User

There is no "identity login" to the MCP. The proof is cryptographic and account-scoped:

1. The browser page is served by the local MCP bridge and fetches a one-time signing request.
2. The request is bound to one plan hash, one account, one network, and one expected signer class.
3. The passkey or local key signs the exact transaction/auth payload.
4. The signed XDR is valid only if it satisfies the smart account's current owner/admin authorization rule.
5. F1 re-reads the account before submit and rejects if the account, signer set, verifier, or rule state changed.

If a different person or device signs, either:

- the signed XDR does not authorize the target smart account and simulation/submission fails, or
- the signer is already an authorized owner/admin for that smart account, which is valid by the account's own policy.

The MCP does not need a password from the user to verify identity. The account's authorization logic verifies the signer.

## 15. smart-account-kit Adapter Contract

Do not spread raw SDK calls throughout the browser app. Create one adapter:

```ts
interface SmartAccountKitAdapter {
  createWallet(input: CreateWalletInput): Promise<CreateWalletResult>;
  connectWallet(input: ConnectWalletInput): Promise<ConnectWalletResult>;
  signPlan(input: SignPlanInput): Promise<SigningResult>;
  signTransaction(input: SignTransactionInput): Promise<SignedTransactionResult>;
}
```

Adapter responsibilities:

- Pin and expose SDK version.
- Normalize account address, signer model, verifier address, and public key data into our schemas.
- Convert E1 plan steps into the SDK's expected transaction/auth-payload format.
- Keep passkey APIs browser-only.
- Return signed XDR only.
- Never return secrets or raw private key material.

MCP responsibilities stay separate:

- Build unsigned XDR.
- Verify signed XDR.
- Submit only through F1.

## 18. Security Checklist

Bridge:

- Bind `127.0.0.1` only.
- Use random port.
- Store only bearer hash.
- Put bearer in URL fragment.
- TTL every request.
- Single-use every request.
- Constant-time bearer comparison.
- Host and Origin checks.
- CORS disabled.
- No secrets in logs.
- No external scripts in companion bundle.
- CSP blocks framing and remote script execution.

Plan verification:

- Signed XDR must match requested unsigned XDR semantically.
- Plan hash must match.
- Step order and step hash must match.
- Network passphrase must match.
- Account address must match.
- Auth entry expiration must still be valid.
- F1 must re-run live preflight before submit.

User safety:

- Owner key never enters MCP.
- Session key risk is shown.
- Revocation plan is always generated.
- Mainnet disabled by default until Phase 9.
- Approval token remains out of model-visible schemas.

## 19. Tests

Unit tests:

- `T-WB.request-token-single-use`: successful result consumes bearer.
- `T-WB.request-expiry`: expired request rejects all endpoints.
- `T-WB.host-check`: wrong Host header rejected.
- `T-WB.origin-check`: wrong Origin rejected.
- `T-WB.no-token-logs`: logs do not contain bearer, approval token, or private-key-like values.
- `T-WB.semantic-xdr-match`: signed XDR with changed operation is rejected.
- `T-WB.step-coverage`: missing or duplicate signed step rejected.
- `T-WB.reject-flow`: browser rejection resolves as rejected, not timeout.

Integration tests:

- `T-WB.browser-smoke`: Playwright opens approval URL, fetches request, rejects.
- `T-WB.sign-plan-mock`: mocked smart-account-kit signs test XDR and callback is accepted.
- `T-F1.1-9`: real testnet install through browser companion and F1.
- `T-WB.readback`: after install, inspect-account sees the new rule.

Security tests:

- `T-WB.csrf-fetch-blocked`: cross-origin POST cannot complete.
- `T-WB.dns-rebind-host-blocked`: hostile Host header fails.
- `T-WB.replay-blocked`: reused callback fails.
- `T-WB.tampered-plan-hash`: browser returns mismatched plan hash and is rejected.

Manual testnet rehearsal:

1. Create/connect smart account in browser companion.
2. Record a simple testnet transfer.
3. Synthesize a recipient+amount policy.
4. Produce proof table with one allow and at least three denies.
5. Sign install in browser.
6. F1 submits.
7. Read back rule.
8. Repeat covered action with session key.
9. Attempt different recipient/amount and confirm fallback to owner approval.
10. Revoke and read back denial.

## 21. Open Engineering Decisions

These must be resolved before coding the adapter:

1. Exact pinned smart-account-kit version.
2. Whether the companion app uses `localhost` or `127.0.0.1` as the WebAuthn RP origin after browser testing.
3. Whether session private key is MCP-held for demo or companion-held from day one.
4. Exact Stellar RPC endpoints and relayer configuration for testnet.
5. Whether browser auto-open is allowed by the MCP host or remains a user-click link.

Recommended defaults:

- Pin smart-account-kit in `package.json`.
- Generate URLs with `127.0.0.1`.
- If passkey RP testing requires it, switch only the approval URL host to `localhost` while still binding loopback.
- Use MCP-held session key for the first demo, clearly labeled as scoped and expiring.
- Keep direct testnet submit first; add relayer after direct path is green.
- Keep browser auto-open optional; plain link must always work.

## 22. Final Invariant

The browser companion can sign, but it cannot decide. The MCP can decide and verify, but it cannot owner-sign. F1 can submit, but only after the disk plan hash chain, live account state, signed XDR, network, and human approval token all match.
